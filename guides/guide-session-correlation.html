Session correlation

Every board entry the agent writes carries a back-link to the session that produced it.

Scope: the session-correlation capability — the footer, the accessor, and the join to traces. Setting tracing up is a separate concern (owned by claudetracing) and is deliberately out of scope here.

opt-in capability board ↔ trace join Claude Code + Copilot _session: <id>_

TL;DR

When the agent opens or comments on an issue, it appends one line — _session: <id>_. That <id> is the harness session id, and it is exactly the id the MLflow Stop hook tags the session’s trace with. So from any board entry you can jump straight to the prompts and tool calls that produced it. The agent learns its own id from $CLAUDE_SESSION_ID, populated once per session by a small SessionStart hook. None of this creates traces — it only links to traces you are already recording.

Why it’s worth doing

Agent-authored issues and PRs normally have no provenance. You can read what was decided, but not how — which prompts, which exploration, which tool calls led there. That gap costs you in three places:

  • Auditability & trust. “An agent wrote this” is only credible if you can open the actual session behind it. The footer turns a board entry into a click-through to the real interaction.
  • Coaching & improvement. The downstream skills — awow-usage-coach, prompt-skill-analysis — assess how people prompt. They can only tie feedback to outcomes if board entries name their session.
  • Digests & reporting. daily-digest and weekly-digest join board activity to session data. Without the id, the two surfaces stay disconnected.
The example you asked for. Because the footer resolves to a full trace, a single issue becomes a worked demonstration of how the prompts and interaction actually unfolded — not a paraphrase, the real thing.

How it works

Two independent tracks run during a session and meet at a shared id. The correlation track (this skill) makes the id available and stamps it on the board. The tracing track (owned elsewhere) writes the trace and tags it with the same id.

The same flow in words

1
Session starts. The SessionStart hook fires once and runs session_env_hook.py, which reads the session_id from its stdin payload and appends export CLAUDE_SESSION_ID=<id> to $CLAUDE_ENV_FILE.
2
The id is now ambient. Claude Code sources that file into every Bash tool call for the rest of the session. It is shell-side, not context-side — zero per-turn cost, and it survives compaction and /clear. The hook re-runs on resume/clear/compact, so it stays populated.
3
The agent writes to the board. When it creates or comments on an issue/PR, the footer rule tells it to read $CLAUDE_SESSION_ID and end the body with _session: <id>_.
4
The trace is recorded in parallel. Independently, the MLflow Stop hook writes the session’s trace and tags it mlflow.trace.session = <id> — the very same id.
5
The join exists. Footer id == trace tag. From the board you reach the trace; downstream skills filter traces by that id to reunite outcome with process.

What is responsible for what

The capability is deliberately small. It does not write traces and it does not configure tracing — it sits on top of a tracing stack that is already running.

How the configuration happens

It is opt-in. On its own the skill changes nothing; a team turns it on during /setup-awow (Step 3) or later via /awow-add. Opting in is gated on tracing already being wired.

0 · Prerequisite check (tracing must already be on)

The skill refuses to proceed if tracing is not wired — otherwise it would stamp footers whose id points at traces that were never written (dead links). It looks in .claude/settings.local.json for both markers:

// .claude/settings.local.json — the tracing markers the check looks for
{
  "env": {
    "MLFLOW_CLAUDE_TRACING_ENABLED": "true",
    "MLFLOW_TRACKING_URI": "databricks://<profile>",
    "MLFLOW_EXPERIMENT_NAME": "/Workspace/Shared/<team>",
    "DATABRICKS_CONFIG_PROFILE": "<profile>"
  },
  "hooks": {
    "Stop": [ { "hooks": [ { "type": "command",
      "command": "uv run python -c \"from mlflow.claude_code.hooks import stop_hook_handler; stop_hook_handler()\"" } ] } ]
  }
}
If the check fails, the skill stops and points you at ../claudetracing to set tracing up first. It will help wire the local settings.local.json against it, but it never owns tracing setup — that stays a separate step.

1 · Wire the accessor hook

Added to the same per-machine settings.local.json, alongside the Stop hook. Takes effect on the next session (or on resume/clear/compact).

// .claude/settings.local.json — "hooks" block, added by opt-in
"SessionStart": [ { "hooks": [ { "type": "command",
  "command": "python3 \"$CLAUDE_PROJECT_DIR/.agents/skills/session-correlation/scripts/session_env_hook.py\"" } ] } ]

2 · Install the footer rule into the team’s conventions

The skill appends Rule 4 to output-discipline.md and a shape note to board-output.md, then re-runs gather.py. The base templates stay clean — the rule lands only for teams that opted in.

## Rule 4 — Session footer
Every board entry you author ends with:

    _session: <session-id>_

Read the id from $CLAUDE_SESSION_ID. It matches the trace's
mlflow.trace.session tag, joining board content to traces.
Exempt: metadata-only changes and one-line status comments.

3 · Verify

  • Start a fresh session, run echo "$CLAUDE_SESSION_ID" — it prints the current id (the same id as the transcript filename under ~/.claude/projects/<project>/).
  • Have the agent post a test comment; confirm the _session: <id>_ line is present and the id resolves to a trace in the team’s experiment.

What it deliberately does not do

Out of scope

  • Setting up MLflow / Databricks tracing — that is claudetracing.
  • Writing traces — that is the Stop hook.
  • Forcing anything into the always-read core instructions.

In scope

  • Exposing the session id to the agent ($CLAUDE_SESSION_ID).
  • The footer convention and where it’s required vs. exempt.
  • The opt-in flow + the tracing prerequisite check.

The footer rule is written harness-neutrally (“your harness’s session id”). The accessor shown here is Claude-Code-specific (SessionStart + CLAUDE_ENV_FILE); other harnesses keep the same footer and supply their own accessor for exposing the session id to the agent.

GitHub Copilot is proven to work. The same footer convention runs under Copilot with its equivalent accessor — we have verified it end to end. That support is landing via a pull request that is still pending, so for now it lives outside the merged template; once merged, Copilot is a first-class harness for this capability alongside Claude Code.

Downstream: what you do with the traces

Correlation is the join. Acting on it is a separate pipeline of analysis skills — the read side. They share only one thing with this guide: they need the traces to exist. Each is covered in its own guide; in brief:

  • mlflow-export — pulls every trace (and the per-session grouping) down to local JSON.
  • prompt-skill-analysis — turns a session (or an export) into a prompt-quality report.
  • awow-usage-coach — reads an export to produce team nudges or individual coaching.
Where the footer pays off. Because each board entry names its session id, these skills can tie their findings back to specific issues and PRs — outcome reunited with process. That is the analysis guide’s subject, not this one’s.
