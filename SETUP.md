# /setup-awow — long-form walkthrough

The wizard is **incremental and resumable**. Each invocation processes one or more markdown files; answers are tracked in `setup-progress.md` at the repo root. Re-invoking `/setup-awow` reads the progress file and picks up where the last session left off.

On every invocation the wizard lays out its full plan — every step from 0 to 10, marked as complete / pending / untouched — so the user always sees the whole map before any individual step runs.

Hard prerequisite: a board with a hierarchy (Linear, Azure DevOps, Jira, or GitHub Issues). The wizard refuses to proceed past Step 1 without one.

## Step 0 — installer (REQUIRED)

Required to make the repo usable. The wizard handles this for you — no separate manual step.

**What the wizard does:**
1. Detects whether `.venv/` exists at the repo root and the pointer stubs in `.claude/commands/` and `.github/prompts/` are populated. If so, marks Step 0 complete and moves to Step 1.
2. Otherwise, asks for explicit permission to run the platform-appropriate installer (`./setup/install.sh` on macOS / Linux, `.\setup\install.ps1` on Windows / PowerShell).
3. Runs the installer. Surfaces its output verbatim. If it fails (most commonly: `uv` not on PATH), surfaces the error and tells the user to install `uv` (e.g. `brew install uv`) and re-invoke `/setup-awow`.
4. Verifies `.venv/` and the pointer stubs are now present.
5. Records Step 0 complete in `setup-progress.md`.

## Step 1 — kickoff (REQUIRED)

The first of the two required steps. After Step 1 the agent can read and propose against the board, and the repo is usable.

**Inputs the user provides:**
- Board URL (`https://linear.app/...`, `https://dev.azure.com/...`, etc.)
- Any preference between Claude Code / GitHub Copilot (auto-detected from `.claude/` / `.github/` if present)

**What the wizard does:**
1. Infers the board tool family from the URL (Linear / ADO / Jira / GitHub Issues).
2. Loads the matching reference from `context/tooling/boards/<tool>.md`.
3. Prompts the user to install the corresponding MCP if it is not wired.
4. Verifies read-write MCP semantics (writes go through `proposals/` first).
5. Writes `context/tooling/board.md` with the team's actual choices.
6. Records Step 1 complete in `setup-progress.md`.

## Recommended next (NOT REQUIRED — any order)

Re-invoking `/setup-awow` offers these in the order they pay back fastest. Any can be deferred. State is tracked in `setup-progress.md`.

### Step 2 — Mission

Output: `context/team/mission.md`.

The single highest-leverage sentence the team writes. Per the blog (§3 of `input/PROPOSAL.md`'s anchor), a vague mission produces vague agent behaviour across every ceremony.

The wizard asks the user for a one-sentence mission and refuses anything trivial ("be excellent", "ship software"). A useful mission names the audience, the change being made, and the constraint.

### Step 3 — Required conventions

Output: `context/team/conventions/REQUIRED/*.md` — issue-titles, labels, branches, output-discipline.

If the board has been used and shows existing patterns, the wizard **observes** and drafts conventions from what it sees. If the team is greenfield, the wizard **guides** them with sensible defaults.

`output-discipline.md` is non-negotiable; without it the agent over-produces from session one.

Optional conventions (`infra-naming.md`, `data-objects.md`, `code-style.md`) appear as `# OPTIONAL — defer` stubs and are not asked about during setup.

### Step 4 — Team members and style

Output: `context/team/members.md`, `context/team/style/*.md`.

Member list with roles, responsibilities, focus areas. Style guidance for board output (tight, score-like), comments, and prose (long-form, complete paragraphs).

### Step 5 — `CLAUDE.md` and `AGENTS.md`

Output: `.agents/CLAUDE.md` regenerated by `tools/bootstrap-claude-md.py` from the stub plus all answers gathered so far. Mirrored to `.claude/CLAUDE.md` and `.github/AGENTS.md` by `tools/gather.py`.

Includes a `## Do not propose` block — explicit scope-shedding ("we are not adding multi-user this quarter"). Stops the agent from helpfully reintroducing scope at session start.

### Step 6 — Knowledge base seed

Output: `context/knowledge-base/README.md` populated; `glossary.md` seeded if the team has terms to copy in.

Glossary, decision template, runbook template. Empty subfolders are tracked by `.gitkeep`.

### Step 7 — Neighbouring teams

Output: `context/company/neighbouring-teams.md`.

1° teams as empty stubs. The convention: each team writes its own summary. The wizard generates stubs the team can fill in later or hand to the neighbouring teams.

### Step 8 — Adoption plan

Output: `adoption-plan.md` at repo root.

Pick the early adopter (using the signals from `input/research/.../adoption_interview_guide.md` if available). Pick the first feature. Record the 4-week Seed schedule.

### Step 9 — Surface the extras

The wizard reads `.agents/commands/{spread,standardise}/` and lists what's available behind `/awow-add`, with phase and prerequisites. Nothing is installed; the user sees what's there.

### Step 10 — Skills review (keep / customise / drop)

The starter pack ships several skills under `.agents/skills/`. Each is opinionated about *some* part of the stack — the harness session format (Claude Code JSONL vs. Copilot vs. other), the tracing backend (Databricks MLflow vs. LangSmith vs. raw files), or the rubric (the seeded user-story template). They will not fit every team out of the box.

The wizard walks each shipped skill once the team has enough context to call it. For every skill: summarise the rubric, name the specific assumption it bakes in (quoted from the skill's own "Starter shape — adjust for ..." callout where present), and ask **keep / customise / drop**. Customisations land under `proposals/setup/step-10/<skill>/` before being moved into the skill's final location. Dropped skills are removed and noted in `setup-progress.md` so the wizard doesn't keep re-offering them.

Re-run this step whenever the stack changes — new harness, new tracing backend, new shared rubric. Skills review is not a one-shot.

## Quickest-quickstart (alternative entry point)

For users who already know what they want and don't need the wizard:

A single copy-paste prompt that performs Steps 0 → 1 → 2 → 3 → 5 in one Claude turn against a fresh repo. Step 0 (installer) still requests permission before running the shell script; the rest skips the per-step review loop. The prompt is generated by `tools/bootstrap-claude-md.py` with `--quickstart` and lives in `.agents/commands/setup-awow.md`'s "Quickest-quickstart" section.

Lower activation energy than the wizard. The wizard remains the recommended path for teams new to the model.

## Proposal-first

Every file the wizard would write lands first as a draft under `proposals/setup/`. The user reviews, then the wizard moves files into their final locations. This is the proposal-first principle from the blog: iterate on cheap-to-change artefacts; only land work when the proposal is right.
