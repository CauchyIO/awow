Transcript router

One entry point. The router reads the transcript, recommends a specialist, and walks you through gated review — meeting to board in minutes.

Phase Spread Command /process-transcript Audience anyone turning meetings into board-ready work
router pattern gated pipeline single entry, many specialists

TL;DR

As your team adopts more transcript-driven prompts (/coaching-review, /solution-design-flow, future leaves), you shouldn’t have to know which one to invoke. One command — /process-transcript — reads the transcript, names what kind of session it is (or which segments of which kinds), recommends one or more specialists with a one-line rationale, and waits at a gate. Confirm, and it dispatches; correct it, or override. The philosophy: talk like a human, let the agent handle the admin. Every meeting becomes structured input that feeds your board, your docs, and your next meeting.

1. The shape — one entry, many specialists

Cognitive load is the enemy. The more specialists a team adopts, the harder it gets to remember which one to type. /coaching-review? /solution-design-flow? When the answer depends on what kind of session you just had, the router should answer it for you.

The router recommends; you type one command/process-transcript

Router / recommender Existing specialist Proposed / in flight Future placeholder

Without a router

You have to know the catalogue. New specialists added → the team has to learn them. Wrong choice → wrong output. The catalogue grows faster than the team’s memory.

With a router

You type one command. The router reads the transcript, names what it saw, recommends specialists. Adding a new specialist is a registry entry, not a re-training event.

2. Two rules the router obeys

Honest routing. Always say what was detected and what was recommended, in one line at the top of the output. Silent dispatching creates “why did it do that?” debugging.
Always overridable. --as=<skill> forces a specific specialist and skips detection entirely; --yes skips the dispatch-confirmation gate when you already trust the call. The router gets it wrong sometimes; you get the last word.

Before you run it · once per repo

3. Set up the repo

Three things, then you never touch them again. Skip this if your repo was bootstrapped with /setup-awow.

  1. The prompt is present. /process-transcript ships under .agents/commands/ and is mirrored to the harness surfaces by tools/gather.py — .claude/commands/ for Claude Code, .github/prompts/ for GitHub Copilot. You invoke it the same way from either harness.
  2. A place to drop transcripts. Keep a working folder for your .vtt files — e.g. .github/transcripts/. It starts empty.
  3. Keep transcripts out of git. Transcripts are working files and may contain personal data — add the folder to .gitignore. The trace pipeline records the path to a transcript, never its contents.
# .gitignore — add at the bottom
.github/transcripts/
Cloud PC users: if your editor runs on a Cloud PC but the recording downloads to your laptop, copy the VTT across (OneDrive, email-to-self, or clipboard paste if supported) before you point the prompt at it.

4. Get the transcript

The router reads whatever file you hand it. Prefer WebVTT (.vtt) over .docx — it preserves speaker tags (<v Speaker Name>) and timestamps, which the router uses for attribution, segment boundaries, and chronological ordering. Plain text/Markdown and .srt work too.

From Microsoft Teams:

  1. Open the meeting (in the recap, or live during the call).
  2. Click the Transcript tab.
  3. Download → choose .vtt (WebVTT).
  4. Save it into your transcripts folder.

Name the file descriptively — refinement-payments-rewrite.vtt beats meeting1.vtt. The filename is a hint the router uses when it classifies the session.

Got an agenda? Paste it alongside the transcript. The router will note which agenda items the meeting actually covered versus skipped. It will not fetch transcripts or agendas for you — you supply the file.

5. Team context sharpens everything

Before touching the transcript, the router reads your team context from context/team/ — members, conventions, glossary, neighbouring teams, the board pointer. This is what turns best-effort guessing into accurate work:

Context fileWhat it buys you
context/team/members.mdSpeaker disambiguation and correct attribution — a garbled name that sounds like a known member resolves to that member.
context/knowledge-base/glossary.mdDomain terms and abbreviations survive voice-to-text mangling.
context/company/neighbouring-teams.mdCross-team blocker detection — dependencies get traced to the right board.
context/tooling/board.mdThe board family and read/write surface, so proposed actions match your hierarchy.

The router never gates on context — missing files just mean a degraded-confidence label, not a stop. If context is stale (>~8 weeks) or absent, it warns and offers /setup-awow, then proceeds with what it has.

The run · a gated pipeline

6. The run — Gate 1: detected & recommended

Open a fresh chat session and point the command at your file. A 30–60 min transcript takes a couple of minutes — the agent reads the whole thing, cross-references names, and classifies content.

/process-transcript .github/transcripts/your-meeting.vtt

Phase 0 → 4, with two gates you control/process-transcript

At Gate 1 the router stops and shows what it heard before it touches a specialist or the board. It looks like this:

GATE 1 — DETECTED & RECOMMENDED

Detected 2 segment(s):
  00:00–00:32  retrospective       (8 participants, confidence: clear)
  00:32–01:01  solution design     (4 participants, confidence: likely)

Recommended dispatch:
  /process-retro          on segment 1  — looking-back framing, peer dynamic
  /solution-design-flow   on segment 2  — options weighed, architecture decided

Duration: ~61 min | Participants: Alex, Sam, Pat, …
Disambiguation: "S-D-W" → SDW (glossary); "Sam" turn at 00:14 likely Pat

What to check here

This is your chance to verify it understood the conversation. Scan for:

  • Session type — did it classify each segment correctly?
  • Participants — are the names right?
  • Decisions vs discussion — did it separate what was decided from what was merely explored?
  • Disambiguation — were garbled words and shared-device mis-attributions corrected sensibly?

If something is off, just say so in plain language — then confirm:

The person labelled "Sam" around 00:14 is actually Pat. And on
segment 2 we landed on option B, not A — A was only explored.
Three ways past Gate 1. Reply go to proceed as shown; --as=<skill> to override a segment’s disposition; or fallback to skip dispatch and run templated extraction across the whole transcript. The router asks at most two clarifying questions, and only when getting an answer wrong would change a board write — otherwise it presents and waits.

7. Gate 2: board actions, or a proposal

After Gate 1, segments that matched a specialist are dispatched — each runs its own pipeline (including its own gates) and owns its own board writes. The router stitches every output into one composite report. If every segment dispatched, you’re done — there is no Gate 2.

Gate 2 fires only when at least one segment fell through to templated extraction. For those fallback action items, the router does board discovery — matching against existing items, detecting cross-team blockers, spotting gaps — and presents a compact action list:

GATE 2 — PROPOSED ACTIONS

Board mapping: 2 matched | 1 new | 1 cross-team dep | 0 untracked

Actions:
  UPDATE  1. #482 — add refinement outcome, move to Ready
  CREATE  1. Story "Add idempotency key to payment retry"
  ESCALATE 1. blocked by Platform #91 → cross-team sync
  KB       1. decisions/payment-retry.md — chose at-least-once + dedupe

Options: "go" · "skip 2,3" · "review" · "cancel"

Here you choose your path. Both are first-class — the router never writes to the board without explicit approval at this gate.

Path A · execute board actions

Board is set up and the work is real? Reply go and the router creates/updates items one at a time, confirming each. Use skip or review to be selective.

Path B · capture to a proposal

Early-stage, pre-requirements, or board hierarchy still fuzzy? Decline the board — “don’t touch the board, drop it all in proposals/<topic>.md” — and keep the detail without committing to structure yet.

When Path B earns its keep: the project is still in discovery or prototype phase; requirements aren’t validated with business users yet; you want to accumulate context across several meetings before the board exists. This is the proposal-first principle — iterate on the cheap artefact, land on the board once it’s solid.

9. Iterate across meetings

The real power is compounding context. After the next meeting, open a fresh session and point the router at both the new transcript and the proposal you saved last time:

/process-transcript .github/transcripts/new-meeting.vtt

# Also attach: proposals/<topic>.md (from the previous session)

The router intersects the prior context with the new conversation — updating decisions, resolving open questions, adding requirements, and flagging contradictions.

Each pass enriches the proposal until it’s board-readycompounding

Voice-to-text shortcut. Answer the generated questionnaire by dictating into your machine (macOS dictation, or Windows voice typing). Save the text and process that as a transcript too — it’s a conversation with yourself, captured.

10. Go agentic — parallel execution

Once you have board items, the workflow shifts from planning to execution. Open one agent session per item and let each pick up its work — /process-workitem takes a single item from refinement to PR.

The board becomes the orchestration layerparallel

You think less about admin and more about direction: the agents handle the 80%, you steer the 20%. That is the agentic way of working — meeting → transcript → structured work → parallel execution → delivered output.

11. Sub-modes live inside leaves

Some session types have variants that share a pattern library. /coaching-review is the clean example: the same prompt handles both 1:1 coaching (pairing, demo, onboarding) and 1:many teaching (course, lecture, cohort). The teacher / learner dynamic is the same; the patterns to watch for differ slightly.

This is a deliberate choice over splitting into separate skills:

  • The shared core (parsing, relationship detection, phase breakdown) lives in one place. No duplication.
  • Adopters learn one skill, not five. The variant gets detected, not declared.

Apply the same idea as new skill types differentiate. When in doubt: sub-mode first, sibling skill only if the work diverges enough that the shared core is no longer shared.
