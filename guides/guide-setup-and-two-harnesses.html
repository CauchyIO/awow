Setup & the two harnesses

What a new adopter hits first: the /setup-awow wizard, and why one set of instructions serves two agents.

Two halves. The wizard turns a fresh clone into a configured repo, one approved step at a time. The pointer-stub model lets Claude Code and GitHub Copilot share a single source of truth without drift. Board specifics live in the companion guide-board-and-mcp.html.

day one incremental & resumable one source, two surfaces

TL;DR

/setup-awow is the wizard you run inside an agent session. It is incremental and resumable — state lives in setup-progress.md, and on every invocation it reads that file and lays out the whole plan (Steps 0–9), marking each ✓ done, ⧗ pending, or ☐ untouched. Only Step 0 (installer) and Step 1 (board kickoff) are required; the rest are recommended-next in any order. Every artefact lands under proposals/setup/ first and only moves to its final path after you approve it. Separately, all agent instructions are authored once under .agents/; tools/gather.py mirrors them into tiny pointer stubs in .claude/ and .github/ so the two harnesses stay in sync and nothing drifts.

Half 1

The /setup-awow wizard — from clone-and-go to a configured repo, one approved step at a time.

How the wizard behaves

The pain it removes is the “I cloned the repo — now what?” problem. Rather than a wall of setup docs, it is a conversational wizard that does the configuring with you. Four properties define how it runs:

  • Incremental & resumable. State lives in setup-progress.md at the repo root. The wizard reads it on every invocation, so you can stop after any step and pick up later exactly where you left off.
  • Always shows the whole map. Before doing anything, it lists every step (0 → 9) and marks each ✓ complete, ⧗ deferred/pending, or ☐ untouched, then tells you which step it is resuming. You never see a step in isolation.
  • Proposal-first. Every artefact is written to proposals/setup/<step>/ first. It moves to its final location (e.g. context/team/mission.md) only after you explicitly approve. The cheap-to-change draft is where you iterate.
  • Two required, the rest optional. Step 0 and Step 1 make the repo usable. Steps 2–9 are recommended-next in any order — the wizard offers the next one and lets you choose.
Multi-workspace runs. /setup-awow --root <path> resolves setup-progress.md, proposals/setup/ and context/ relative to <path>/ instead of the repo root — useful when one repo hosts several workspaces. The harness infrastructure (.venv/, .agents/, .claude/, .github/) and the installer stay at the repo root regardless; Step 0’s detection inherits the parent repo’s installer state.

The --quickstart path. /setup-awow --quickstart does Steps 0 → 1 → 2 → 3 → 5 in one turn with sensible defaults, skipping the per-step review loop. Step 0 still asks permission before running the shell installer. It is the one-shot path for adopters who already know what they want; the conversational wizard remains the default.

Step 0 — Installer required

The starter pack uses tools/gather.py to mirror .agents/ into the harness surfaces. Step 0 wires Python via uv, creates .venv, and runs gather once — which is what makes /setup-awow itself discoverable. It starts with a deliberately cheap detection probe so the wizard stops scanning the moment the answer is obvious:

Both present

.claude/commands/setup-awow.md and .venv/ both there — Step 0 is already complete. Skip ahead.

venv missing only

Stubs present, no .venv/. Gather has run; only the env needs restoring. The wizard offers uv sync --python 3.12 — not the full installer — after confirmation.

Stubs missing

Run the full installer: ./setup/install.sh (macOS / Linux) or .\setup\install.ps1 (Windows).

In every branch it requests permission before running any shell installer and surfaces the output verbatim — it never runs the script silently. The most common failure is uv not being on PATH; the wizard surfaces that error and tells you to install uv first, rather than papering over it by guessing a system Python. It then verifies .venv/, .claude/commands/setup-awow.md, and .github/prompts/setup-awow.prompt.md are all present.

Step 1 — Board kickoff required

The outcome of Step 1 is a wired read/write surface plus a fully-populated context/tooling/board.md — the team’s actual board spec, not just MCP wiring. The agent reads that file thereafter whenever it needs to know what a label means, which states are terminal, or where a new issue belongs. It runs in two parts:

Step 1a · wire the surface

Detects or installs the read/write surface — an MCP for Linear / Jira / Azure DevOps / GitHub, or the gh CLI for GitHub-hosted boards. Verifies read with one call and write with a no-op write against a scratch issue. A surface that cannot finish this session is recorded pending so the repo is still partially usable.

Step 1b · configure

Mode chosen automatically by counting closed issues. Mode A (<10 closed) sets up from the reference — accept / override / skip per decision. Mode B (≥10 closed) assesses and captures what is already on the board, recording divergence from the reference.

Review-and-adjust gate. Once board.md is landed, the wizard does not silently move on. It reads the file back, summarises it, and asks whether to proceed, adjust a section, or evaluate it against the live board — looping until you say proceed. Full board mechanics (MCP install, label taxonomy, the gh CLI alternative) live in the companion guide-board-and-mcp.html.

Half 2

One source of truth, two harness surfaces — author once, let gather.py mirror it so nothing drifts.

The trap, and awow’s answer

A team using both Claude Code and GitHub Copilot hits the same problem: every instruction file, prompt, and skill ends up duplicated across .claude/ and .github/ — and the two copies drift. Someone fixes a convention in one place, forgets the other, and now the two agents follow different rules.

awow’s answer is pointer stubs. You author everything once under .agents/. Then tools/gather.py generates tiny redirect files in .claude/ and .github/ that each harness discovers natively. A stub carries only the discovery metadata the harness needs — frontmatter description / name — plus a one-line body that says “read .agents/… and follow it.” There is no substantive content in a stub, so there is nothing to drift.

Source of truth Generator Generated stubs (per harness)

.claude/ and .github/ are committed so a fresh clone is immediately recognisable to either harness. Because stubs carry no substantive content, the only thing gather keeps in sync is discovery metadata — and --check catches that automatically.

What a stub actually contains

Everything substantive stays in .agents/. The generated file is just frontmatter the harness uses for discovery plus a one-line redirect — so even if you read it, it points you straight back to the source:

<!-- .claude/commands/refinement-prep.md — GENERATED by tools/gather.py. DO NOT EDIT. -->
---
description: draft a feature for the next refinement
---
Read .agents/commands/refinement-prep.md and follow it.
One Copilot gotcha worth knowing. Prompts under .github/prompts/ must end in .prompt.md — VS Code’s Copilot Chat silently ignores a plain .md there. gather.py emits the right extension automatically; you only ever need this fact when debugging “my new command isn’t showing up in Copilot.”

Regenerating & keeping it honest

After you edit anything under .agents/, regenerate the stubs. In CI, the same script with --check fails the build if a stub is out of date — the drift you were trying to avoid is now caught mechanically.

# edit the source of truth
$EDITOR .agents/commands/refinement-prep.md

# regenerate the .claude/ and .github/ stubs
uv run python tools/gather.py

# in CI: fail if any stub drifted from .agents/
uv run python tools/gather.py --check

The rule of thumb: edit under .agents/, never the generated stubs. Hand-edits to .claude/ or .github/ are overwritten on the next gather. The same mechanism carries the CLAUDE.md that Step 5 produces out to .claude/CLAUDE.md and .github/AGENTS.md.

Why this matters for adopters

The recommended way to adopt awow is GitHub’s Use this template, not a fork — so your repo starts with no upstream relationship to merge through. The starter ships both .claude/ and .github/ populated, which means a fresh clone is recognisable to either agent out of the box. The wizard’s Step 1 detects which harness you are actually running inside and asks whether the team also uses the other, so the source-of-truth promise scales whether you run one harness or both.
