Transcript router

One entry point. The router recommends; the user types one command.

Scope: how /process-transcript routes a transcript to a specialist. This is not a catalogue of every command — the full list lives in .agents/commands/README.md.

router pattern recommender behaviour single entry, many specialists

TL;DR

As your team adopts more transcript-driven prompts (/process-retro, /coaching-review, /solution-design-flow…), the user shouldn’t have to know which one to invoke. One command — /process-transcript — reads the transcript, identifies what kind of session it is (or which segments of which kinds), recommends one or more specialists with a one-line rationale, and waits for confirmation. The specialists stay focused; the user gets a single front door.

Why a router and not separate commands?

Cognitive load. The more specialists a team adopts, the harder it gets to remember which one to type. /coaching-review? /process-retro? /solution-design-flow? When the answer depends on what kind of session you just had, the router should answer it for you.

Without a router

User has to know the catalogue. New specialists added → team has to learn them. Wrong choice → wrong output. The catalogue grows faster than the team’s memory.

With a router

User types one command. Router reads the transcript, names what it saw, recommends specialists. Adding a new specialist is a registry entry, not a re-training event.

Two rules the router obeys

Honest routing. Always say what was detected and what was recommended, in one line at the top of the output. Silent dispatching creates "why did it do that?" debugging.
Always overridable. --as=<skill> forces a specific specialist. The router gets it wrong sometimes; the user gets the last word.

Sub-modes live inside leaves

Some session types have variants that share a pattern library. /coaching-review is the clean example: the same prompt handles both 1:1 coaching (pairing, demo, onboarding) and 1:many teaching (course, lecture, cohort). The teacher / learner dynamic is the same; the patterns to watch for differ slightly.

This is a deliberate choice over splitting into separate skills. Two reasons:

  • The shared core (parsing, relationship detection, phase breakdown) lives in one place. No duplication.
  • Adopters learn one skill, not five. The variant gets detected, not declared.

Apply the same idea as new skill types differentiate. When in doubt: sub-mode first, sibling skill only if the work diverges enough that the shared core is no longer shared.

Fallback — when no specialist fits

When inference is ambiguous, or the transcript doesn’t match any specialist’s shape, the router stays in /process-transcript and produces its existing templated output (actions, owners, follow-ups). No specialist = no surprise output. The router degrades gracefully into the behaviour the team already knew.
