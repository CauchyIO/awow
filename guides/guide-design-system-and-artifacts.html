A design system is a durable team artifact. You build it once (/design-system); then every deck, blog, and solution design is rendered from it plus a content file (/artifact). Opt-in — board-only teams never touch it.

1. What a design system is in awow

Not a token JSON file and not a Sass package. In awow a design system is a small set of cooperating artifacts, all hand-readable and self-contained:

ArtifactWhat it is
Style guide
style-guide.html		One self-contained HTML page with the tokens as CSS variables and live component demos. This is the source of truth. Versioned; old versions deprecated explicitly.
Templates
templates/<type>/		One template.html + TEMPLATE.md per artifact type (presentation, blog, one-pager). The HTML carries the full token block; the markdown carries a content→component map so an artifact can be generated without re-deriving anything.
Pointer
context/tooling/design-system.md		Says whether a system exists and where. Every artifact-producing command reads it first. Covered in §2.
The rule
CLAUDE.md		An enforced instruction: when an artifact is produced and a system is present, read the source and adopt it — never invent styling.
Why one self-contained HTML page? Open it anywhere, mail it, print it to PDF — no build step (this page works that way). Cost: tokens duplicate across files; the "re-read the source" rule keeps them from drifting.

2. The pointer — one place that knows

context/tooling/design-system.md ships as mode: absent. That single field is what lets the rule "follow the design system if present" mean something everywhere — a command reads the pointer and either enforces or stands down.

The three modespointer

stand down system in this repo system in a sibling repo

One rule rides on every mode: always re-read the source before generating — the pointer's token summary is a cache and can drift. For a private external repo, read it by filesystem path (a board MCP will 404).

Rare · once per brand

3. Establishing it — /design-system

A gated pipeline. Stop at each gate, show the work, wait for approval. You build the thing artifacts are generated from — you do not generate a deck here.

Establish — three gates/design-system

3.1 Tear down a reference, empirically

Capture a reference site's computed styles with a headless browser — colours, fonts, sizes, tracking — not just screenshots. Extract the method, then re-express it in your own tokens. Don't clone a palette and call it a system.

3.2 Define a small token set

Resist a sprawling palette — the load-bearing set is short:

#f6f5f3background
#ffffffsurface
#f24b37accent — logo only
#1a1a1atext 1
#6b6b6btext 2
#9a9a9atext 3

One background, one surface, one accent (logo or single emphasis — never a fill), a three-step text ramp, named semantic tints, a spacing scale. Pick fonts, cap the weight. The swatches above are awow's own — yours will differ.

3.3 Write the guide, build templates, wire the pointer

Write the self-contained style-guide page with live demos; build template.html + TEMPLATE.md per artifact type; set the pointer. Render it in a browser before the final gate — never ship a guide you haven't seen.

Frequent · every artifact

4. Producing an artifact — /artifact

Lighter than establishing, but it still gates on content before generating — regenerating HTML from changed markdown is cheap; rewriting hand-tuned HTML is not.

Apply — content first, render last/artifact

1 · Content first

Draft and agree the markdown. Gate before any HTML exists.

2 · Generate from template

Keep its <style> and nav verbatim. HTML/CSS diagrams over raw SVG or Mermaid.

3 · Verify, then export

Check layout in a headless browser; PDF via Chrome headless (modern CSS survives); fix overflow, regenerate.

4 · Land it

Commit markdown + HTML + PDF; update the board.

No system yet? If the pointer is absent, /artifact offers /design-system first, or proceeds with plain defaults on your say-so — it never fakes a house style.

5. How it plugs into the rest of awow

The design system is not a silo — the "adopt it if present" rule reaches every surface that emits HTML.

SurfaceWhat it does with the design system
CLAUDE.md rule"When you produce an HTML artifact" — read the pointer, adopt the system when present, re-read the source each time. Applies even to ad-hoc HTML that never goes through /artifact.
/solution-design-flowPhase 0 reads the pointer; the presentation track adopts the system when present and hands the generate-and-render mechanics to /artifact rather than duplicating them.
/setup-awow Step 8Detects an existing system; if none, asks one question — "do you produce styled HTML artifacts?" — and, on yes, suggests /awow-add design-system. Implicit and opt-in; never auto-run.
DigestsStyled HTML digests adopt the same tokens, so a daily/weekly digest looks like it belongs to the team rather than to the tool.

6. Conventions that make it work

Accent is the logo, full stop

The single accent colour is reserved for the wordmark or one emphasis. Never a background fill. Hierarchy comes from weight, not colour.

Borders over shadows

1px borders and a small radius, not drop shadows. Calmer, prints cleanly, reads the same on every surface.

Semantic tints

Each tint family means something (plan, action, reference). Colour carries information; it is not decoration.

Content before HTML

Agree the markdown, then generate. Never hand-author a styled artifact when a template exists.

Re-read the source

The pointer's token summary is a cache. The style guide is authoritative; re-read it each run so artifacts don't drift.

Self-contained, single file

Inline CSS, inline SVG, no build step. Open it anywhere, print it to PDF, mail it. Like this very page.

For each team to answer

7. Open questions a team should answer

Calls each adopting team makes for itself — intentionally not decided here.

Q1. In-repo or external? Keep the system under context/design-system/ for self-containment, or point at a dedicated design repo you already maintain? External wins when design assets live elsewhere; in-repo wins when they don't.
Q2. Which artifact types earn a template? Every type you produce more than once or twice. A presentation template almost always; a one-pager template only if one-pagers recur.
Q3. Render toolchain. Chrome headless is the default for print-to-PDF because modern CSS survives. Is that available in your environment, or do you need a different exporter (and to constrain the CSS accordingly)?
Q4. Cache vs source drift. The pointer caches a token summary. How hard do you enforce "re-read the source"? A lint that diffs the cache against the style guide, or trust the rule?
Q5. Should awow itself declare its own system? These guides clearly use one, yet the shipped pointer is absent so adopters start clean. If the project dogfoods its own pointer, keep adopters resetting to absent on clone.
