Session timeline

An interactive visual timeline of a project's Claude Code sessions, built straight from the raw logs — no tracing setup required.

The zero-setup, visual on-ramp to awow's session-analysis story. This is the picture that trace analysis describes in prose — but it reads Claude Code's own JSONL, so there is nothing to wire up first.

zero setup Claude Code only vanilla JS + SVG timeline.html

TL;DR

tools/session_timeline.py reads the raw session logs Claude Code already writes to ~/.claude/projects/<encoded-path>/*.jsonl and emits two files: sessions.json (the parsed data) and a fully self-contained timeline.html (vanilla JS + SVG, no server, no CDN). The page is a Gantt swimlane of every session — coloured by functional area, packed so concurrent runs stack — with a concurrency strip, handoff arrows, idle gaps, and per-session peak context. Click any bar for a detail panel; optional flags fold in coach reviews and a whole-project overview. Its one advantage over the MLflow path: it needs no MLflow, no Databricks, no Stop hook — just the logs on disk.

No tracing required. The trace-analysis path (mlflow-export → prompt-skill-analysis / awow-usage-coach) needs traces to already exist in MLflow, written by a Stop hook. This tool reads the JSONL Claude Code writes for you anyway. If the logs are on disk, you can render the timeline — that is the whole point.

What the timeline shows

The page is one tall SVG read top to bottom. Each element answers a different question about how the work actually unfolded:

ElementWhat it reveals
Gantt swimlaneOne bar per session. Width is duration, colour is the functional area (the top-level dir most touched), and bars are packed into rows so overlapping sessions stack rather than collide. Read-only / research sessions are drawn hatched.
Concurrency stripA band across the top counting how many sessions ran simultaneously at each moment — the awow fan-out / contract rhythm made visible.
Handoff arrowsAn arrow from an earlier session to a later one when the later session continued the earlier one's files (inferred per-file last-writer).
Idle gapsPink bands where no session logged any event — the true gaps between activity, separating active time from elapsed time.
Peak contextPer session, the input + cache tokens at its fullest turn, plus output tokens, with a bar. Sessions past the standard 200K window are flagged.
Detail panelClick any bar to open a panel for that session; it can embed a coach review and the project overview sits there by default.

The swimlane and the concurrency strip

The swimlane is the spine of the page. Sessions are placed on a real wall-clock x-axis and packed greedily into rows: a session takes the first row whose last bar ended before it starts, so two sessions that overlapped in time end up on different rows. The result reads like a Gantt chart of a day's work, and the row count at any vertical slice is the parallelism.

  • Colour = functional area. Each session is bucketed by the top-level directory it touched most (e.g. tools/, ios/, docs/), so a glance at the palette shows which part of the codebase each burst of work was in.
  • Hatching = read-only. A session that only read / searched and wrote nothing is drawn with a hatch fill, so research and reconnaissance are visually distinct from sessions that changed code.
  • The concurrency strip sits above the lanes and steps up and down with the live count of open sessions. Its shape is the story: work fans out into several parallel short sessions, contracts back to one, then fans out again. That cadence is awow's core rhythm, and here you can see it instead of describing it.
Why packing matters. Without row-packing, concurrent sessions would draw on top of each other and the parallelism would be invisible. Packing is what turns a flat list of sessions into a picture of fan-out.

One evening, or many days

The swimlane above is the view of a single day. When a project's sessions span more than one day — common for trace exports that cover weeks or months — the page switches automatically to calendar mode:

  • A day-navigator strip runs along the top, one cell per active day, its height scaled to that day's session count. Long empty stretches between active days are collapsed to a small // 12d marker, so a four-month project stays a few screens wide instead of hundreds.
  • Click a day, or press ←/→, to load that day's full detail below — the same swimlane + concurrency strip + idle gaps, scoped to that day's hours. The selected day is marked in the brand accent.
  • Sessions resumed across days (a session id that keeps living for a week) are placed on their start day and clipped at midnight with a → +Nd marker, so one long-lived id never stretches the whole axis.
Whole-project shape up top, hour-level detail below. The navigator is the overview — sparse weeks vs. crunch days at a glance — while each day still renders at full resolution when you open it.

Handoffs and idle gaps

Handoff arrows

The tool tracks, per file, which session last wrote it. When a later session edits a file an earlier session had touched, it draws an arrow from the earlier bar to the later one — an inferred handoff. Across a project this surfaces the threads of continuity: which session picked up where another left off, even when they were days apart and never shared a transcript.

Idle gaps

Idle bands are computed from event spacing, not from session open/close times. A gap is idle when no session logged any event during it. This is deliberate: a session left open while the human is away is not counted as active just because it is technically running — if nothing was logged, the band is pink. The contrast between the coloured bars and the pink bands is the difference between active time and elapsed time.

Read idle gaps as questions, not waste. A pink band might be a coffee break, a meeting, or a long human review — the timeline tells you where the day went, not whether that was good or bad.

Per-session peak context

For each session the tool finds its fullest turn and records the input + cache tokens at that point (peak context) alongside total output tokens, shown as a bar in the detail panel and on the row. Peak context is the better health signal than a running total: it is how close that session came to filling its window.

  • Sessions whose peak context exceeded the standard 200K window are flagged — a cue that the session was operating in compaction / large-context territory, where quality and cost both shift.
  • Reading peak context next to duration and area helps spot the sessions that were doing heavy, context-hungry work versus quick, focused ones.

Detail panel, coach reviews, and the overview

Clicking a bar opens a detail panel for that session: its area, duration, peak context, file footprint, and handoff links. Two optional inputs enrich it:

--coach-dir

A directory of per-session coach reviews (markdown). When a review matches a session, it is embedded directly in that session's panel — so the prose assessment sits next to the bar it is about.

--overview

A whole-project overview / meta-analysis markdown file. It is embedded as the default panel — what you see before clicking any bar — framing the whole timeline.

Both are markdown you bring; the tool only renders and places them. That keeps the picture (this tool) and the prose (coaching skills) cleanly separable while letting them appear in one view.

How to run it

One command. Point it at the repo, give it an export of the transcripts to read, optionally fold in coach reviews and an overview, set your timezone, and choose an output dir.

python tools/session_timeline.py \
    --project-path <repo> \
    --transcripts ./_exports \
    --coach-dir ./coach_reviews \
    --overview ./OVERVIEW.md \
    --tz-offset 2 \
    --out .
FlagPurpose
--project-pathThe repo whose sessions to render; used to locate the encoded log dir under ~/.claude/projects/.
--mlflow-exportAlternative source. An mlflow_export/ dir of Databricks traces (from mlflow-export) instead of local logs. Supports multiple users.
--userOptional. Scope an MLflow export to one user (matches mlflow.user, local-part or email).
--transcriptsAn export directory of the session JSONL to read.
--coach-dirOptional. Directory of per-session coach reviews (markdown) to embed in panels.
--overviewOptional. A project overview / meta-analysis markdown to use as the default panel.
--tz-offsetHours offset from UTC, so the wall-clock axis reads in your local time.
--outWhere to write sessions.json and timeline.html.

To read Databricks traces instead of local logs, swap the source flag:

python tools/session_timeline.py \
    --mlflow-export ./mlflow_export/<experiment> \
    --user casper \
    --tz-offset 2 \
    --out coach_reviews/<repo>-retrospective
Then just open it. timeline.html is self-contained — double-click it. No build step, no local server, no network. sessions.json stays next to it for any other tool that wants the parsed data.

Where it fits among the analysis tools

This is the visual, zero-setup on-ramp to awow's session-analysis story — the picture that guide-trace-analysis.html describes in words. It complements the other tools rather than replacing them:

ToolRole
session timeline (this)The picture. Reads raw Claude Code JSONL or an MLflow trace export, renders the interactive timeline. No tracing infrastructure needed for the local-log path.
session correlationThe board↔trace plumbing — how a board entry gets linked to a recorded trace. Different concern; this tool does not touch it.
awow-usage-coach / prompt-skill-analysisTrace-driven prose coaching and prompt-craft scoring. Embed coach markdown here via --coach-dir; for a deep read, defer to those skills.
Two sources, different fidelity. The local-log path (Claude Code only) carries the most detail — file edits, per-turn context, handoff arrows. The MLflow path works for any traced harness and supports multiple users, but traces carry no file paths (bars colour by working directory) and no per-turn context. Copilot teams use the MLflow path.
