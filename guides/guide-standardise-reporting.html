Standardise reporting

The read-only synthesis layer: a day, then a week, then the neighbouring teams — each a different altitude on the same activity.

Three commands in the Standardise layer. /daily-digest and /weekly-digest are live; /cross-team-view is documented but not yet built (planned for v0.3). All three only read — they never touch the board or the code.

standardise read-only synthesis day → week → cross-team cross-team-view: v0.3

TL;DR

Once a team is past the Seed stage and most people are committing, the board lists what changed but nobody can see the shape of it. The Standardise reporting layer answers the harder question — what actually happened, where is it heading, what connects, and what should someone know that they don’t? /daily-digest does this for one day and writes digests/YYYY-MM-DD.md; /weekly-digest does it for a Mon–Fri window at a higher altitude and writes digests/weekly/YYYY-Www.md; /cross-team-view is the planned third altitude for neighbouring teams — a stub today, real in v0.3. None of them mutate anything.

When this layer makes sense. Reporting is a Standardise-stage concern: it earns its keep only once there is a steady stream of activity to synthesize. The shared prerequisites are that the team has shipped at least three Seed cycles, most of the team is actively committing, and Step 0 of /setup-awow is complete (the agent can read and write the board). /weekly-digest adds one more: /daily-digest must have been running for at least four working days, because the dailies are its richest input.

The three altitudes

Same underlying activity — board, code, and (optionally) chat — observed from three heights. As you climb, the question changes: from “what happened today?” to “what shifted Monday-to-Friday?” to “what did the team next door ship?” The week is not a stack of dailies; it answers different questions, even though it reuses their collection and synthesis machinery.

The one principle that runs through all of it

The board already gives you a list of changes. A digest that just re-lists them adds nothing. The job is synthesis, not aggregation — turning raw activity into the few things a busy teammate actually needs:

  • What actually happened? A narrative, not bullets.
  • What’s the trajectory? Which projects are moving, stalling, or blocked.
  • What connects? Work by one person that relates to another’s — dependencies and overlaps that aren’t formally tracked.
  • What should someone know that they don’t? Cross-relevance the board would never surface on its own.
Be specific or stay silent. “A’s rate-limit work could inform B’s gateway redesign” is useful. “Everyone should stay aligned” is noise. Per-person takeaways follow the same rule — only genuinely relevant signals, and an empty section is fine. Don’t force relevance.

/daily-digest — one day, synthesized

Collects the day’s board, code, and (optional) chat activity, synthesizes it, and writes digests/YYYY-MM-DD.md. The flow is a short pipeline:

Step 0 ─ Input & mode detection   (markdown-only, or email mode)
Step 1 ─ Data collection          (board + code + optional chat)
Step 2 ─ Synthesis                (the four questions above)
Step 3 ─ Markdown output          → digests/YYYY-MM-DD.md
Step 4 ─ (optional) HTML render   → GATE: manual review in browser
Step 5 ─ (optional) Email send    (only after explicit approval)

What it collects

  • Board activity — every issue touched today: who, what changed, which project. If the team has a private surface (e.g. a leadership-only board), that data is pulled internally to inform per-person sections but is excluded from anything shared.
  • Code activity — commits, PRs, reviews, mapped back to board issues where a branch or commit message names one. Repos active without a ticket are a gap signal worth noting.
  • Chat messages (optional) — channel messages only, where a channel-to-project mapping exists. Meeting transcripts are always excluded from automatic ingestion — they carry personal data and are handled separately by /process-transcript.

An empty result on a day you know was busy is treated as a probable query fault (wrong team name, stale credentials), not as truth. “skip chat” and “skip code” are honoured; the digest produces from whatever returned.

What it produces

A markdown file with a data-sources table, a day-at-a-glance metrics table, a team narrative, per-project status snapshots, cross-team connections, code activity, per-person takeaways, and structural observations (gaps, stale issues, untracked dependencies). Every board identifier it cites must correspond to an issue actually seen during collection — counts are never fabricated.

Email mode has a hard gate. If — and only if — recipient addresses are supplied, the digest renders to HTML from digests/TEMPLATE.html, opens it in the browser, and waits for explicit approval before sending. The review gate is mandatory and never skipped; recipients are never hardcoded. No emails, no markdown is ever shared without confirmation.

/weekly-digest — the same thing, a week up

This is /daily-digest run over a Monday–Friday window — not a summary of the dailies. It reuses the daily command’s collection, synthesis discipline, and behavioural boundaries unchanged, then asks different questions at a higher altitude. Output is digests/weekly/YYYY-Www.md (e.g. 2026-W12; no argument means the current ISO week).

What carries over from the daily, untouched

  • The same board / code / chat surfaces and the same sync-resilience handling.
  • Synthesis, not aggregation; cross-relevance detection; genuinely-relevant per-person sections only.
  • Read-only, data-grounded, no performance evaluation, no strategic recommendations, private-team data strictly excluded from the shared digest.

What changes at the weekly altitude

  • An extra primary input: the week’s daily digests. It reads every digests/YYYY-MM-DD.md in the window — these are its richest source. A missing daily is a data-coverage gap to flag, not a day to silently skip.
  • A comparison against last week. If digests/weekly/YYYY-W(ww-1).md exists, it reads it to detect dropped connections and compare trajectories.
  • Weekly shapes instead of single-day snapshots. Counts (issues created / completed / stale, active projects, PRs merged); collaborations classified as active (appeared 3+ days), emerging (first this week), or dropped (active last week, absent this week); and a trajectory — where each project stood Monday vs. Friday and its direction of travel.

The output adds week-specific sections: a project trajectory report (Monday vs. Friday with a direction), a team activity heatmap (relative effort per person per project), the three-way collaboration breakdown, and a personalized week-in-review. Delivery follows the daily pattern — including the same render → review-gate → send flow if the team uses email mode, never sending without approval.

Why not just summarize the dailies? Because a week answers questions a day cannot: what moved (outcomes, not activity), what shifted Monday to Friday, where the team spent its time, and which patterns are emerging. Altitude, not aggregation.

/cross-team-view — the planned third altitude

stub — documented, not implemented ships in v0.3
Not yet built. /cross-team-view is a stub. It is documented as the next altitude — neighbouring-team visibility, the “the team next door shipped what?” problem — but the working version ships in v0.3. Today it has no behaviour to describe, and this guide deliberately doesn’t invent any.

What exists today is the opt-in path, not the command. Running /awow-add cross-team-view reports the command’s prerequisites (the same Standardise gate: three Seed cycles shipped, most of the team committing, board read/write working). Once those are met and the command is real in v0.3, the stub file becomes the prompt the agent actually runs — the same pattern the digests already follow. Until then it is a placeholder marking where the third altitude will sit.

The boundaries that hold across all of them

Read-only, always

No command here mutates the board, the codebase, or any external system. The only writes are the digest files under digests/ — and, in email mode, an HTML render that is never sent without approval.

Data-grounded, never evaluative

Synthesize only from real board / code / chat data. Never fabricate or misattribute. Never evaluate individual performance. Surface connections; let humans decide — no strategic recommendations.

Private data stays private

Private-team surfaces never appear in shared digests. They’re pulled internally only to inform per-person sections for people who already have access.
