# Proposal: maintainer meta-instructions (`MAINTAINING.md`)

**Status:** PARKED 2026-05-25 — deprioritised as a minor, non-critical guide-sync tweak. The board-discipline half is handled by the `meta/` workspace proposal; the taxonomy/pillar-sync half is parked. Revisit only if the prompt catalogue grows enough that the guide drifts in practice.
**Author:** agent (drafted with casper)
**Date:** 2026-05-25

## Problem

awow wears two hats:

- **Product** — `.agents/CLAUDE.md` is a bootstrap stub shipped to adopters via "Use this template", regenerated by `tools/bootstrap-claude-md.py` once they run `/setup-awow`. Written for someone *using* awow.
- **Project** — the rules for *building awow itself* (keep the prompt-taxonomy guide in sync, how a new branch is added, awow's own board discipline).

Today there is no surface for the second hat. The maintainer team works in the repo root and the agent loads only the shipped stub — i.e. the maintainers are running on the adopters' manual. Putting project rules into the stub fails twice: every new adopter inherits maintainer trivia, and the content sits at the wrong layer (bootstrap may regenerate around it).

Decision (confirmed): the prompt taxonomy is a **project concept**, not a product concept. It is coupled to this repo's `guides/` and `.agents/commands/` layout and has no adopter-side meaning.

## Relationship to the `meta/` workspace proposal

A parallel proposal (`meta/proposals/meta-workspace-and-fixture-decoupling.md`) promotes `dogfood/` → `meta/` as awow's real awow-on-awow workspace, read via `--root meta/`. That changes how directives split, because two surfaces load at different times:

- **Harness-loaded session rules** — top-level `.agents/CLAUDE.md`. Loaded for *every* session regardless of `--root`. Fires when you edit a `.md` under `.agents/commands/`.
- **`--root meta/` context** — read *only* when an awow command is invoked against `meta/`. `--root` is an awow-command convention, not a harness one.

Therefore:

- The **board-led / proposal-first** half is **subsumed by the `meta/` workspace** (its board config points at `CauchyIO/awow`). This proposal does **not** duplicate it.
- The **taxonomy-sync rule cannot live in `meta/`** — editing a command file is an ordinary session where `meta/context/` is never read. It must stay on the harness-loaded top level.

This proposal therefore narrows to: the taxonomy-sync rule + the undocumented "adding a new pillar" gap.

## Approach

| Kind | Location | Why |
|---|---|---|
| Rule the agent follows while editing commands | `.agents/MAINTAINING.md` (new) | harness-adjacent top level; gather mirrors it; bootstrap never touches it |
| Durable rationale / definitions | `context/knowledge-base/` | what a pillar / specialist / sub-mode *is* |
| Board discipline for awow's own work | the `meta/` workspace | already its purpose — not duplicated here |

`.agents/MAINTAINING.md` is **not auto-loaded** by the harness — only `CLAUDE.md` is. So the shipped stub gets a single pointer line; that one breadcrumb is the only maintainer trace adopters inherit, and it is what makes the rule actually fire for maintainers.

## Terminology to settle first

The taxonomy guide uses **router → specialists**, not "pillar":

- **Router / recommender** — `/process-transcript`, the single entry point.
- **Output families** (top-level branches) — *Action-output*, *Pattern-output*, *Specialized signal*.
- **Specialists** (leaves) — `/coaching-review`, `/solution-design-flow`, …
- **Sub-modes** — variants inside one specialist (e.g. 1:1 vs 1:many coaching).

**Terminology decision (recommended): define "pillar" = output family.** The three families (*Action-output*, *Pattern-output*, *Specialized signal*) are co-equal top-level categories that specialists slot under — exactly what "pillar" connotes. Use "pillar" in prose; keep "branch" only for the router *tree diagram*. Align the guide to this once agreed.

So: **"adding a new specialist"** = a leaf (already documented in the guide: five artefacts → `python tools/gather.py` → update the phase `README.md`). **"adding a new pillar"** = a new top-level output family — **undocumented today**, and the real gap this proposal closes.

## Proposed content of `.agents/MAINTAINING.md`

> Working agreement for developing awow itself. If you are *using* awow on your
> own team, you do not need this file — read `CLAUDE.md`.

### Prompt taxonomy stays in sync

- When you **add or rename a specialist** (a command under `.agents/commands/`),
  follow the "Adding a new specialist" checklist in
  `guides/guide-transcript-router.html` and update that guide's diagram/table in
  the **same change**. The filesystem is the runtime registry; the guide is the
  human onboarding view and must be synced by hand.
- When you **add a new pillar** (output family beyond Action-output /
  Pattern-output / Specialized signal), follow the "Adding a new pillar"
  runbook in `context/knowledge-base/<entry>` and reflect it in the guide.
  Name every file a maintainer must touch.

Board discipline for awow's own work lives in the `meta/` workspace, not here
(see the meta-workspace proposal).

## Proposed pointer line in `.agents/CLAUDE.md`

The `MAINTAINING.md` rules link to it; without this entry the "how a branch is added" rule points at nothing.
