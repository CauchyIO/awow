Trace analysis

What you do with the sessions once they exist: export, then assess.

Companion to the session correlation guide. That one is the write side — how an entry gets linked to a trace. This one is the read side — turning traces into prompt-quality and adoption reports.

read side export → assess three skills, one dependency

TL;DR

One skill pulls the traces down to local JSON (mlflow-export); two skills read that JSON to produce markdown reports — prompt-skill-analysis (how well someone prompts) and awow-usage-coach (how a team or person works, and what to change). In every case the script only counts and extracts; the qualitative judgment lives in the agent.

The one shared prerequisite. All three need the traces to exist. They do not record sessions — that is the MLflow Stop hook, set up via the claudetracing library. If the footer convention is also on (see the session correlation guide), these reports can tie their findings back to the exact issues and PRs a session produced.

The three skills

Step 1 · pull

mlflow-export

Paginates the experiment and writes every trace (info, request, response, spans) plus a per-session grouping to local JSON.

In: experiment id or URL + Databricks profile.

Out: traces.jsonl, sessions/<id>.json, manifest.json.

Step 2a · assess prompting

prompt-skill-analysis

An evidence-backed read on how someone prompts: clarity, specificity, structure, iteration, voice, with concrete suggestions.

In: an mlflow_export/ dir or a raw session JSONL. Auto-detects single vs. multi-session.

Out: prompt_skill_report.md.

Step 2b · assess working

awow-usage-coach

Reads sessions through three lenses — intent shape, sequence patterns, edit footprint — and reports in one of two modes.

In: an mlflow_export/ dir. --user sets the subject.

Out: a team-nudge or self-coach report.

How awow-usage-coach reads a session

It is vocabulary-agnostic on purpose — it works whether or not a team uses awow's exact slash commands, because it measures behaviour, not labels. Three lenses, in priority order:

LensWhat it measures
Intent shapeEvery prompt is classified into one of eight labels: investigate, plan, propose, implement, refine, verify, document, inform (else other).
Sequence patternsBigrams / trigrams of intent transitions per session reveal the working rhythm (investigate→implement→verify, propose→implement→refine…). In self-coach mode, subject vs. team rhythms are compared.
Edit patternsEach trace's files.modified is bucketed by type (proposal, context, agents-config, code, markdown…) and crossed with intent — e.g. “when teammates 'propose', 70% of touched files are .md; when you 'propose', 50% are code.”

team-nudge mode · no --user

Proposes additions to .agents/CLAUDE.md / copilot-instructions based on recurring patterns across the team.

self-coach mode · with --user

Imperative, encouraging coaching for one developer, compared against the team baseline.

Running the pipeline

Export once, then point either reader at the result. The scripts are plain Python next to each skill.

# 1. pull the traces down to JSON
.venv/bin/python .agents/skills/mlflow-export/scripts/mlflow_export.py \
    --experiment-id <ID> --profile <PROFILE> --out ./mlflow_export

# 2a. prompt-quality report (export dir OR a raw session JSONL)
python3 .agents/skills/prompt-skill-analysis/scripts/extract_prompts.py \
    --input ./mlflow_export --out ./analysis.json
#     then the agent reads analysis.json and writes the markdown report

# 2b. usage coaching — omit --user for team-nudge, add it for self-coach
python3 .agents/skills/awow-usage-coach/scripts/awow_extract.py \
    --input ./mlflow_export --out /tmp/awow_usage.json [--user <email>]
Each is just a skill. In practice you invoke them by name (/mlflow-export, /prompt-skill-analysis, /awow-usage-coach) and the agent runs the extractor, then does the qualitative write-up. The commands above are what runs underneath.

Backend & harness portability

These ship as starters for Databricks MLflow + Claude Code. The analysis rubrics are harness-agnostic — only the input parsing is tied to a shape. A team on another backend either emits the same JSON layout that mlflow-export produces, or extends the extractor scripts (extract_prompts.py, awow_extract.py) with a reader for their format. That customisation is exactly what /setup-awow Step 9 (Skills review) is for.
