◆ Agentic Transcript Workflow

Welcome

From Meeting to Board in Minutes

This guide walks you through the agentic transcript workflow — a method for turning raw meeting recordings into structured proposals, board items, and architectural artefacts using GitHub Copilot in VS Code.

Meeting ─→ VTT transcript ─→ Copilot processes ─→ Proposal.md ├─→ Board items ├─→ Architecture diagrams └─→ Questionnaires

The philosophy: talk like a human, let the agent handle the admin. Every meeting becomes a structured input that feeds your project board, your documentation, and your next meeting — creating a compounding loop.

This workflow works with any project. You only need VS Code, GitHub Copilot (Agent mode), and a meeting transcript.

Step 1

Set Up Your Repository

Required

Before you process your first transcript, your repository needs three things:

  1. A .github/prompts/ folder — contains process-transcript.prompt.md, which is the engine behind the workflow. Copy from the awow starter pack or your team's existing prompt library.
  2. A .github/transcripts/ folder — this is where you drop your .vtt files. It should be empty to start.
  3. A .gitignore entry — add .github/transcripts/ to your .gitignore. Transcripts are working files; they should not be committed to the repository.
# .gitignore — add at the bottom
.github/transcripts/

Cloud PC users: If your VS Code runs on a Cloud PC but your recording downloads to your laptop, you'll need to copy the VTT file across. Use OneDrive, email it to yourself, or paste via the clipboard if your Cloud PC supports it.

Step 2

Download Your Transcript

Required

After your meeting ends (or during — it's available live):

  1. Open the meeting in Microsoft Teams.
  2. Click the Transcript tab (in the meeting recap or during the call).
  3. Click Download → choose .vtt format (WebVTT).
  4. Save the file into your .github/transcripts/ folder.

The .vtt format preserves speaker tags (<v Speaker Name>) and timestamps, which the processor uses for speaker attribution and chronological ordering. Prefer VTT over .docx.

Name the file descriptively — use the meeting title or a short label like refinement-payments-rewrite.vtt. The filename helps Copilot classify the meeting type.

Step 3

Process the Transcript

Required

Open GitHub Copilot Chat in VS Code (the side panel). Start a new session — a clean context produces better results.

Type the prompt command and point it to your transcript file:

/process-transcript C:\path\to\.github\transcripts\your-meeting.vtt

Or attach the prompt and file manually:

# In Copilot Chat, attach:
#  1. The prompt file:  .github/prompts/process-transcript.prompt.md
#  2. The transcript:   .github/transcripts/your-meeting.vtt
# Then just press Enter.

Copilot will now:

  1. Load the team context file (if it exists)
  2. Parse the entire VTT — disambiguate speakers, fix garbled names
  3. Classify the meeting type (standup, refinement, 1:1, architecture discovery, etc.)
  4. Extract structured content using the matching template
  5. Present Gate 1 — a summary for you to review

This takes time. Processing a 30–60 min transcript may take 1–3 minutes. The agent is reading the full transcript, cross-referencing names, and classifying content.

Step 4

Review Gate 1 — The Summary

Required

Copilot stops at Gate 1 and presents a structured summary. This is your chance to verify it understood the conversation correctly:

  • Meeting type — did it classify correctly? (e.g. "Ad-hoc 1:1" vs "Refinement")
  • Participants — are names correct?
  • Decisions vs discussion — did it distinguish what was decided from what was merely explored?
  • Disambiguation — check that garbled words were corrected reasonably
  • Action items — are they attributed to the right people?

If anything is off, tell Copilot:

"The person you labelled as 'Sam' is actually Pat. Also, we
decided on option B, not A — the discussion was exploring A
but we landed on B at the end."

Once you're satisfied, confirm and Copilot moves to Gate 2 — board proposals.

Step 5

Choose Your Output Path

Required

At Gate 2, Copilot proposes board actions (create/update work items). You have a choice:

Path A — Execute board actions.
If your project is mature enough and you have a board set up, say "go" and Copilot will create/update work items directly.

Path B — Save as proposal.md.
If the project is early-stage (e.g. pre-requirements, prototype phase), tell Copilot:

"I don't want to proceed with the board. Just drop everything in a proposal.md file that I can use later."

This captures all the detail without committing to board structure yet.

When to use Path B:

  • The project is still in discovery / prototype phase
  • You haven't validated requirements with business users yet
  • You want to accumulate context from multiple meetings first
  • The board doesn't exist yet or you're not sure about the hierarchy
Step 6

Enrich the Output

Optional — Recommended

Once you have your proposal or board items, continue the conversation to extract more value. The agent still has the full meeting context loaded.

Ask for clarifying questions:

"Add clarifying questions at the bottom — what's still
missing to execute this project in its entirety?"

This produces a list of unknowns and gaps — great prep for your next meeting.

Ask for a questionnaire:

"Tomorrow I'm meeting with the business. Help me prepare
a questionnaire based on what we discussed — questions
where I really need their input."

Copilot generates 20–30 targeted questions you can bring to stakeholders.

Ask for architecture diagrams:

"Based on our conversation, create an architectural
diagram of the data flows we discussed."

Copilot produces Mermaid or ASCII diagrams that visualize what was talked about.

Ask for wireframes:

"Create wireframes for the UI components we discussed."
Step 7

Iterate Across Meetings

Optional — Power Move

The real power comes from compounding context across multiple meetings. After your next meeting:

  1. Download the new transcript (VTT)
  2. Open a new Copilot session
  3. Point the processor at both the new transcript and the existing proposal.md
/process-transcript .github/transcripts/new-meeting.vtt

# Also attach: proposal.md (from the previous session)

Copilot will intersect the previous context with the new conversation — updating decisions, resolving open questions, adding new requirements, and flagging contradictions.

Meeting 1 ─→ proposal.md (v1) ↓ Meeting 2 + proposal.md (v1) ─→ proposal.md (v2) ↓ Meeting 3 + proposal.md (v2) ─→ proposal.md (v3) ─→ Board items

Each iteration enriches the proposal. When you're confident in the requirements, then push to the board.

Voice-to-text shortcut: You can also answer the generated questionnaire by dictating into your machine (macOS dictation, Windows voice typing with Win+H). Save the text, and process that as a transcript too. It's like having a conversation with yourself — and the agent captures it all.

Step 8

Go Agentic — Parallel Execution

Optional — Advanced

Once you have board items (Azure DevOps work items or GitHub Issues), the workflow shifts from planning to execution:

Board ├── Item A ──→ Agent Session 1 (working) ├── Item B ──→ Agent Session 2 (working) ├── Item C ──→ (not started) └── Item D ──→ Agent Session 3 (working)
  1. Open multiple Copilot sessions — one per board item
  2. Point each session at a specific work item: 
    "Pick up work item PROJ-1042 and implement it."
  3. Let them work in parallel — each agent is autonomous and tracks its own progress on the board
  4. When you lose track — just look at the board. It reflects reality because the agents update it as they work.

The board becomes your orchestration layer. You think less about admin and more about direction. The agents handle the 80% — you steer the remaining 20%.

This is the agentic way of working: meeting → transcript → structured work → parallel agent execution → delivered output.

Appendix A

Team Context File

Optional — Improves Accuracy

A team-context.md file (placed next to the prompt) tells Copilot about your team, your sprint, your vocabulary, and your board structure. This dramatically improves disambiguation and board matching.

# team-context.md
Last updated: YYYY-MM-DD

## Team Members
| Name        | Role              | Focus Areas              |
|-------------|-------------------|--------------------------|
| Alex M.     | Data Scientist    | Internal AI, NLP         |
| Sam K.      | Data Engineer     | Platform, Governance     |

## Current Sprint
Sprint 12 (Apr 21 – May 2)
- Goal: Prototype internal AI dashboard

## Glossary
- SDW: Source Data Warehouse
- BDW: Business Data Warehouse
- DAB: Databricks Asset Bundle

## Dependent Teams
| Team           | Owns                 |
|----------------|----------------------|
| Platform Team  | Catalogue, CI/CD     |
| BI Team        | Reporting layer      |

Without this file, the processor still works — but speaker disambiguation and board matching will be best-effort.
