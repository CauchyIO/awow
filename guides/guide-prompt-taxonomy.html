Prompt taxonomy

One vocabulary, two directions: eight intent labels you prompt with, and the same eight that /awow-usage-coach reads your sessions back through. Naming the move forward is what lets the coaching land backward.

This is the shared language under the Individual tier — the thing that ties writing a prompt to seeing yourself in /awow-usage-coach, the digests, and the session timeline.

Individual tier 8 intents vocabulary-agnostic

TL;DR

awow can read your sessions back to you — /awow-usage-coach classifies every prompt you wrote and reports the patterns. This is the vocabulary it uses, and the trick is that you use the same vocabulary forward: every prompt is doing one of eight things (investigate, plan, propose, implement, refine, verify, document, inform). Name which one before you type and two things happen at once — the prompt gets sharper now, and the reflection later is in a language you recognise. One vocabulary, both directions, is the whole idea.

The loop this closes

This guide is one vocabulary running in two directions. Forward, it’s a prompting discipline you apply in the moment. Backward, it’s the lens the awow tooling uses to reflect your work to you. The value isn’t either half alone — it’s that they’re the same eight words, so the loop closes.

Forward — you, prompting

Name the intent in your first line. If you can’t name it, default to investigate. Two minutes of clarity here saves twenty of meander.

Backward — the agent, reflecting

/awow-usage-coach classifies every prompt in your traces into the same eight, then reports the sequence and edit patterns. Self-coach mode does this for you alone; team-nudge does it across the team.

Why one vocabulary both ways matters. If the discipline you prompt with and the report you read back use different words, the feedback loop never closes — you can’t recognise yourself in the analysis. One vocabulary, both directions, is what makes the reflection actually land.

That also places this guide in the wider set. It sits in the Individual tier — the craft of one session, before any board exists — alongside the session timeline and trace analysis guides. Those show you what a session looked like; this names the moves inside it. None of the three touch the team’s board directly: they earn their place by making the person doing the work sharper.

Why name the move at all

A vague prompt produces a vague session. The cheapest move that fixes this is also the smallest: in your first line, say what kind of move you’re making. The intent name primes the agent’s disposition (verify primes test-running; propose primes drafting-not-applying) and forces you to commit to a posture instead of letting the session drift.

Drifting

can you look at the auth module
and tell me what's going on
maybe also suggest a fix?

Named intent

investigate: auth module — list every
caller of verifyToken() and note any
that skip the expiry check.

(then I’ll plan the fix.)
The diagnostic. If you can’t pick a label for what you’re about to ask, default to investigate — the inability to name the move is the signal that you don’t yet know enough to plan.

The eight intents

Eight labels, plus an other bucket for prompts that don’t fit. The labels are vocabulary-agnostic — they work whether or not the team uses any awow slash-commands.

IntentWhat you’re doingOpening that fits
investigateGathering information; reading code; mapping the terrain.“find every place we call X / explain how Y is wired”
planSketching what you’d do, without committing to a change.“what would it take to add Y? list the steps”
proposeDrafting the change to review before doing it — the awow staple.“draft a proposal in proposals/Y.md”
implementMaking the change in the canonical files.“apply that proposal to Y.ts”
refineIterating on something just made — tightening, polishing, narrowing.“tighten the error message; keep the API the same”
verifyChecking it works — tests, types, the real app, a screenshot.“run the tests / open it and confirm Z renders”
documentRecording the why, for the future reader (often you).“add a why-comment / write a context/knowledge-base/ entry”
informTelling the agent something — a status, decision, observation.“FYI Linear says X is in-progress; don’t recreate the ticket”

The labels are deliberately coarse. They’re posture, not procedure: a single implement can edit twenty files; a single investigate can read fifty. The point isn’t classification accuracy — it’s self-awareness about what kind of move you’re making.

The rhythms that recur

Intent sequences — pairs and triples of intents through a session — tell you more than any single label. A few patterns recur enough to be worth naming:

Healthy rhythms

  • plan → propose → implement — the textbook awow rhythm. Cheap iteration in markdown, then apply once it’s right.
  • implement → verify → refine — TDD-shaped. Healthy when the change actually lands.
  • investigate → plan — you read the terrain, then commit to a route.

Worth nudging on

  • implement → implement → implement with no verify — execution without checks. Tends to correlate with refine later (rework).
  • investigate × 3+ — deep exploration. Fine on unfamiliar code; expensive when the answer was already in context/knowledge-base/.
  • inform-heavy — you’re talking to the agent rather than directing it. Common in pair-style work; reactive when there’s no plan to anchor it.
  • other > 30% — the taxonomy is under-capturing how you actually talk. The gap itself is the signal.

The small ritual

Open every session with a one-liner that names the intent. Two examples that cost nothing and change everything:

plan: what would it take to add cancel-on-blur to the
search input — list the touch points, no code yet.
verify: run the auth tests and tell me which fail
on the new token-expiry branch.

That’s the whole craft. The label is for you first (it forces clarity); the agent picks it up second (and adjusts posture); the coach reads it back third (and the loop closes).

When “other” takes over

If a sizable fraction of your prompts don’t map cleanly to the eight, the taxonomy isn’t broken — your style isn’t the median style the taxonomy was tuned for, and that’s also signal. Two responses both work:

  • Let it be visible. The coach reports it honestly (“other > 30% — the taxonomy under-captures”) and quotes a handful of those prompts. You decide whether to nudge your phrasing or accept it.
  • Don’t over-classify. Some prompts genuinely don’t fit (very short follow-ups, conversational acknowledgements). Misclassifying them just makes the report wrong.
